# Passline

Passline is a free-events SaaS (Eventbrite-style, but events are free) for organizers who
need RSVPs, a waiting list, and door check-in without payments or attendee accounts. An
organizer signs in, creates and publishes an event with a capacity, and shares a public link.
Attendees RSVP with just a name and email; once capacity is reached, new RSVPs land on a
waitlist automatically.

The headline feature is the **Live Waitlist Autopilot**: when a confirmed attendee cancels,
the freed seat is instantly offered to the next person on the waitlist via a time-limited
claim link. If they do not claim it in time, a background sweep re-offers the seat to the
next waitlister -- no organizer intervention required.

## Tech stack

- [TanStack Start](https://tanstack.com/start) (React 19, file-based routing via TanStack
  Router, SSR)
- [Convex](https://convex.dev) for the backend: database, queries/mutations/actions,
  scheduled functions, and cron jobs
- [Convex Auth](https://labs.convex.dev/auth) (Password provider) for organizer sign-in
- [shadcn/ui](https://ui.shadcn.com) + Tailwind CSS v4 for UI
- [Resend](https://resend.com) (via `@convex-dev/resend`) for transactional email
- [`qrcode`](https://www.npmjs.com/package/qrcode) for ticket QR codes
- [Vitest](https://vitest.dev) + [`convex-test`](https://www.npmjs.com/package/convex-test)
  for backend unit tests

## Prerequisites

- Node.js compatible with the pinned package manager below (no `.node-version` file is
  committed; any recent LTS Node works)
- [pnpm](https://pnpm.io) -- this project is pinned to `pnpm@11.8.0` (see `package.json` ->
  `packageManager`). **Use pnpm only**: do not use `npm install`, it will rewrite
  `pnpm-lock.yaml` and break `pnpm install --frozen-lockfile` in CI/deploys.
- A [Convex](https://convex.dev) account (the free tier is enough for local dev)

Install dependencies:

```bash
pnpm install
```

## Running locally

Passline needs two processes running side by side, in two terminals:

```bash
# Terminal 1 -- Convex backend (functions, database, dev deployment)
npx convex dev
```

```bash
# Terminal 2 -- the web app
pnpm dev
```

The app is served at `http://localhost:3000`.

The first time you run `npx convex dev`, it provisions a Convex dev deployment and writes
`CONVEX_DEPLOYMENT`, `VITE_CONVEX_URL`, and `VITE_CONVEX_SITE_URL` into `.env.local` for you
(see `.env.example` for what each of these is). `.env.local` is gitignored and must never be
committed.

## Environment variables

See `.env.example` for the full list with comments. In short:

- **Client (`.env.local`, written by `npx convex dev`):**
  - `VITE_CONVEX_URL` -- Convex deployment URL, read in `src/router.tsx`. The app throws on
    boot if this is missing.
  - `VITE_CONVEX_SITE_URL`, `CONVEX_DEPLOYMENT` -- also written automatically; not read
    directly by application code beyond the Convex client tooling.

- **Server (Convex deployment env vars -- set via `npx convex env set` or the Convex
  dashboard, *not* in `.env.local`):**
  - `RESEND_API_KEY` (`convex/email.ts`) -- enables live transactional email (confirmation,
    waitlist, and claim-link emails). **Deferred if unset**: the email actions become a
    clean no-op instead of throwing, so local dev, tests, and previews work without it.
  - `APP_URL` (`convex/email.ts`, `convex/waitlist.ts`) -- base URL used to build links
    inside emails, e.g. `${APP_URL}/claim/<token>`.
  - `JWT_PRIVATE_KEY` / `JWKS` -- Convex Auth's signing keys, generated by running
    `npx @convex-dev/auth`. **Deferred if unset**: sign-in (`/login`, the organizer
    dashboard) will not work end-to-end without them, but the rest of the app and the test
    suite are unaffected.

## Flows

- **Organizer creates and publishes an event** -- sign in at `/login`, create an event at
  `/events/new` (`convex/events.ts` -> `createEvent`), then publish it (`publishEvent`) so
  it becomes visible at its public slug URL, `/e/$slug`.
- **Attendee RSVPs** -- on `/e/$slug`, an attendee submits name + email
  (`convex/rsvps.ts` -> `rsvp`). While the event has free capacity (derived live via
  `countSeatsTaken`, never stored) the RSVP is `confirmed` and a confirmation email is sent;
  once capacity is reached, new RSVPs are `waitlisted` instead, with a waitlist-position
  email. Attendees manage/view their RSVP at `/rsvp/$token`.
- **Waitlist Autopilot** -- cancelling a confirmed RSVP (`cancelRsvp`) frees a seat and
  immediately calls `promoteNext`, which promotes the next waitlisted attendee to
  `confirmed_pending_claim` and emails them a claim link (`/claim/$token`) with a 30-minute
  window (`CLAIM_WINDOW_MS`). Claiming (`claimSpot`) confirms the seat. If the window lapses
  unclaimed, a 1-minute cron (`convex/crons.ts` -> `sweepExpiredClaimsNow`) sweeps expired
  holds, sends that attendee back to the waitlist, and re-offers the seat to the next person
  in line -- fully automatic, no organizer action needed.
- **QR ticket + live door check-in** -- a confirmed attendee's `/rsvp/$token` page shows a
  QR ticket (via `qrcode`). The organizer's door dashboard at `/events/$id/door` scans/enters
  a token to call `checkIn`, which is concurrency-safe (a duplicate scan of an
  already-checked-in ticket is a harmless no-op) and only checks in `confirmed` seats. The
  dashboard reads `getDoorState` for a live confirmed/checked-in count and a recent
  check-in list.

## Testing

```bash
pnpm test        # run the Convex test suite once (vitest run)
pnpm test:watch  # watch mode
```

Tests use `convex-test` to exercise the real Convex functions (schema, events, RSVPs,
waitlist promotion/claim/expiry, email scheduling, and check-in) without a live deployment.
